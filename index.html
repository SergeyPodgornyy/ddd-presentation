What is the Domain?

  1. may be better known to many developers as the business logic
  2. is "a sphere of knowledge or activity", for instance the business the company runs

Principles behind the DDD

To design a good software, it's important to know what that software is about.

Mainly, domain driven design focuses on:

  1. The core domain and domain logic.
  2. Complex designs on models of the domain.
  3. Improving the application model and resolving emerging domain-related issues by collaborating with domain experts.

Application architecture: Layers of DDD

There are 3 main layers you should know while developing your application:

  1. Domain Layer, that contains the infrastructure independent domain logic.
  2. Infrastructure Layer provides the technology dependent artifacts, like implementation to handle database requests and responses, or remote third-party services calls.
  3. Application Layer acts as a gateway to business logic with integrated transaction control.

Common terms used for DDD

  1. A Domain is a sphere of knowledge.
  2. An Ubiquitous Language is used by all the team in order to connect all the activities of the team with the software.
  3. A Bounded Context is usually the solution space, where we design our software or business solution.
  4. A Context Map displays the alignment of Domains and their Bounded Contexts.

Building blocks

  1. An Entity
  2. A Value Object
  3. An Aggregate (collection of objects)
  4. A Repository
  5. Domain Events
  6. A Service
  7. A Factory

Characteristics of a strong domain model

  1. Being aligned
  2. Being isolated
  3. Be loosely designed
  4. Being reusable
  5. Be an abstract and cleanly separated layer
  6. Minimum dependencies on infrastructure frameworks

Advantages of DDD

  1. Eases Communication
  2. Business necessities oriented
  3. A balanced application
  4. Keeping track made easy
  5. Improves Flexibility
  6. A common set of terms and definitions used by the entire team
  7. Emphasizes Domain Over Interface

Disadvantages of DDD

  1. Requires Robust Domain Expertise
  2. Encourages Iterative Practices
  3. Ill-Suited for Highly Technical Projects
